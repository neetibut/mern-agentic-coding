The task

From a vague ask
to a clear spec.

The same feature, asked two ways — only one is safe to build.

Start with the vague version

The prompt: “Add a way for users to pick a main address.” Reasonable to a human — but here’s what it leaves the agent to guess.

1What does “main” mean — a flag, a field, an ordering?
2Can there be more than one at a time?
3What happens to the old main when a new one is picked?
4What about the very first address?
5What if the user deletes the one that’s main?
6Must existing address behaviour keep working?

Turn it into a spec

Vague request
“Pick a main address”
The agent invents the behaviour, the data model, and the edge cases.
→
Engineering-ready spec
Default Address
Behaviour, constraints, acceptance criteria — and the rule that must always hold.
A good spec isn’t longer — it makes the engineering thinking explicit: behaviour, constraints, and what must not change.

What the spec makes explicit

Behaviour
What it does
First address auto-defaults · marking a new default unsets the old · deleting the default promotes another · the default is badged and listed first.
Constraints
What’s off-limits
No new auth or roles · no external services or dependencies · stay within model → controller → routes and the existing Neon* components.
Acceptance
How we know it’s done
Exactly one default whenever there’s ≥1 address — verified in the browser and by calling the API directly.
The heart of the feature
Exactly one default — enforced on the server.

At most one default, and exactly one whenever there’s at least one address. The rule lives in the controller, not just the UI — a client could call the API directly.

The edge cases the plan must handle

The invariant has to survive every path, not just the happy one.

1The first address addedauto-default
2Switching default between two — the old one clearsexactly one
3Deleting the current defaultpromote another
4Deleting the last addresszero, no error

Ask for a plan — then review it

The agent proposes; the human reviews before any code changes. What I’m checking:

1Is the data-model change minimal — ideally one boolean?minimal
2Is the invariant enforced server-side, not just the UI?correctness
3Are all four edge cases handled?completeness
4Does it touch anything it shouldn’t — auth or profile?scope
Only when the plan passes these do I approve the edit.

Implement narrowly — then verify

Don’t trust “it works.” Walk the behaviour in the browser, then check the rule directly.

1Add the first address — is it automatically the default?
2Add a second — does the first stay default? Make the second default — does exactly one flip over?
3Refresh — did it persist to the database?
4Delete the default — is another promoted? Delete the last — no default, no errors?
5Call the API directly — still exactly one default, without the UI’s help.
The check most people skip is the last one — the invariant holds even when the UI isn’t involved.

The reusable workflow

The same loop works for any change, on any real codebase.

1Context — have the agent inspect before it edits.
2Plan — review scope, data model, and edge cases before approving.
3Implement — narrowly, one logical change.
4Verify — behaviour in the browser, and the invariant directly.
5Review — you stay responsible for correctness.
Not prompting harder — engineering more clearly.
