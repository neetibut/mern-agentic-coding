10-Minute Intro · Teaching Session

Agentic Engineering
for MERN Developers

Not just asking AI to write code — working with an AI coding agent in a disciplined way: clear context, good judgement, review checkpoints, and ownership of the result.

with Neeti

The idea that guides everything
The agent helps us move faster — we stay responsible.

The AI can accelerate the work, but the developer still owns the direction, quality, and correctness of the software.

First

Why MERN?

A practical place to learn the whole picture of full-stack development.

MERN, in four parts

One JavaScript stack, front to back.

MongoDB
The database — where data is stored.
Express
The backend framework — handles requests.
React
The frontend — runs in the browser.
Node.js
The runtime — JavaScript on the server.
Why this stack
A practical on-ramp to full-stack.

MERN exposes you to the whole picture — user interfaces, APIs, databases, project structure, debugging, and maintenance.

Not always the best production stack — but a strong foundation for practising the engineering ideas.

Second

What is
Agentic AI Coding?

Powerful — and exactly where the risk begins.

The definition
Using an AI agent to help build software.

Tools like Claude Code, Cursor, or Codex work inside the codebase — reading files, planning, editing code, running commands, debugging, and writing docs.

The risk
“Build me a full-stack app” makes the AI guess.

One vague prompt, and the agent has to invent everything you didn’t say.

Five things it has to guess

1The users
2The product scope
3The architecture
4The database design
5What “done” means
Impressive on the surface. Fragile underneath.

The result can look great yet be hard to understand, hard to debug, and hard to maintain. That gap is what Agentic Engineering closes.

Third

Coding vs
Engineering

The distinction to remember — one is a tool, the other is a discipline.

Two different things

The activity
Agentic AI Coding
Using an AI agent to help build software.
The discipline
Agentic Engineering
The process that keeps that work clear, reviewable, and maintainable.

Engineering is a way of working

Not a single prompt, not a specific tool.

1Define the problem clearly
2Give the agent useful context
3Break the work into small tasks
4Ask for a plan before code changes
5Review the output and test the behaviour
6Stay responsible for the final result

The useful contrast: Vibe Coding

Vibe coding
No map
A vague instruction — and you accept whatever it produces.
→
Agentic Engineering
Map + checkpoints
The problem, the users, the MVP, the constraints, the architecture — and a plan before any file changes.
Don’t let the AI drive without a map.

Vibe coding hands over the wheel. Agentic Engineering gives the agent a map, checkpoints, and engineering standards.

The first layer

Prepare Context
Before Coding

Before a single line of the app — set the agent up to help well.

What we’d build first: the foundation

The code comes later. This comes first.

    api/    Express API
  packages/    shared code (when needed)
  docs/    ← the part that matters most early on
    BUSINESS_PROBLEMS.md
    PRD.md
    AGENTIC_ENGINEERING_PLAN.md
    ARCHITECTURE.md
  AGENTS.md    ← lean, always-loaded agent context
  README.md
  package.json    workspace root
  .gitignore
The key insight
The most important folder is docs/ — not the code.

Those documents are how you give the agent context. And context is one of the most important materials we provide: the better the context, the better the agent can help.

Keep one lean AGENTS.md at the root as the always-loaded index — it points the agent to these heavier docs instead of repeating them.

Step 1
Start with business problems.

Write BUSINESS_PROBLEMS.md yourself, first. The human developer sets the initial direction — don’t hand this to the AI immediately.

They don’t need to be perfect — just clear enough to review.

Step 2
Use AI to challenge your thinking.

Not a code generator yet — a thinking partner. Ask it to find unclear problems, question your assumptions, and sharpen the statements. No PRD yet.

Good AI-assisted development starts with clearer thinking, not code.

Step 3
Now generate the PRD and architecture.

With the problems clearer, let the agent draft the Product Requirements Document — then the architecture.

What a PRD captures

▸Product overview
▸Target users & their problems
▸Goals & non-goals
▸User stories
▸MVP scope & success criteria
▸Assumptions & open questions

What the architecture doc explains

▸Frontend & backend structure
▸Database design
▸API structure
▸Data flow
▸Key technical decisions
▸Risks & trade-offs
Review — don’t rubber-stamp
The best beginner architecture is the one you can explain.

If the MVP is too big, shrink it. If the design is too complex, simplify it. If the agent adds tools you don’t need, challenge them.

Before any code

Safety Rules

Guardrails that keep you in control of the agent.

Keep control of the work

Plan before code
Ask for a plan before allowing any changes.
Keep tasks small
Never “build the whole app in one prompt.”
Review every change
Read the file diffs before accepting them.
Commit checkpoints
Save working states with Git.

Protect secrets, verify output

Never paste secrets
No API keys or tokens in prompts.
Use .env + .gitignore
Keep secrets out of the repo entirely.
Ask for trade-offs
Make the agent explain its important decisions.
Run & test it yourself
Don’t trust “it works” — confirm it.
The golden rule
Don’t accept code you can’t explain.

You don’t need every advanced detail — but you should know what it’s for, where it lives, what it connects to, and how to check whether it works.

The key lesson
Not AI replacing the developer — the developer with leverage.

Agentic Engineering keeps you in charge of direction, quality, and correctness while the agent does more of the work.
